# Rules — Ridy Agents

이 파일은 agents 레포의 문서 작성 규칙과 컨벤션을 정의합니다.

---

## 마크다운

### 구조
- **H1(`#`)은 파일당 1개** — 파일 제목으로만 사용
- **H2(`##`)부터 섹션 분할** — 논리적 계층 유지
- **H3(`###`)는 세부 항목** — H2 아래에만 사용
- **H4 이하는 피하기** — 구조가 깊어지면 파일 분리 고려

### 코드 블록
- 언어 명시 필수: ` ```typescript`, ` ```json`, ` ```bash`
- 파일 경로 표시: ` ```typescript:path/to/file.ts`
- CLI 명령은 ` ```bash` 사용

### 표
- 헤더 행은 `---`로 구분
- 열이 5개 이상이면 세로 표로 전환 고려
- 설명 열이 길면 마지막에 배치

## 에이전트 스펙 문서 (spec/)

### 작성 규칙
- 각 에이전트는 `### 역할`, `### 담당 레포`, `### 핵심 책임`, `### 작업 방식`, `### 사용 스킬`, `### 제약사항` 섹션 포함
- 핵심 책임은 표 형식 (책임 | 설명)
- 작업 방식은 번호 매기기 (1. 2. 3.)
- 제약사항은 불릿 리스트

### 업데이트 시 체크
- [ ] 다른 에이전트에 영향을 주는 변경인지 확인
- [ ] AGENT_PROTOCOL.md와 일관성 유지
- [ ] docs 레포의 참조 경로 업데이트 필요 여부

## 협업 프로토콜 문서 (protocol/)

### 작성 규칙
- 작업 흐름은 코드 블록 내 번호 매기기
- 에이전트 규칙은 에이전트별 H3 섹션
- 이슈 매핑은 표 형식

### 업데이트 시 체크
- [ ] 작업 흐름 변경 시 모든 에이전트에 영향 없는지 확인
- [ ] GitHub Project 필드/이슈와 일치하는지 확인
- [ ] AGENT_SPEC.md와 충돌하지 않는지 확인

## 개발 기획서

### 위치와 네이밍
- 표준 구현 계획서는 `docs/planning/implementation/YYYY-MM-DD_<repo><issue>-<short-name>.md`에 작성
- 에이전트 프로토콜/스펙/작업 큐 자체 변경 계획만 `agents/plans/PLAN-<순번>-<기능명>.md`에 작성
- `agents/plans/`의 순번은 001부터 순차 증가
- 기능명은 한글 하이픈 케이스 또는 짧은 영문 kebab-case

### 필수 섹션
1. **개요** — 배경, 대상, 기대 효과
2. **기능 분해** — 하위 기능 + 우선순위 테이블
3. **코드 구조** — 백엔드/프론트엔드 구조
4. **상세 설계** — 정상 흐름, 함수 시그니처, 예외 처리, 엣지 케이스
5. **구현/테스트 케이스 등록표** — 기능 코드 항목별 케이스 ID, 구현 파일/단위, 테스트 파일/이름, A/E/X 연결
6. **테스트 시나리오** — UT/IT/E2E
7. **의존성** — 선후관계, docs 업데이트 항목

### 구현/테스트 케이스 등록표 형식
| 케이스 ID | 기능 코드 항목 | 구현 대상 파일 | 구현 단위 | 대응 테스트 파일 | 테스트 이름 | A/E/X 연결 |
|---|---|---|---|---|---|---|
| FE-CASE-001 | 홈 검색 라우팅 | app/page.tsx | handleSearch | __tests__/Home.test.tsx | 경로와 시간을 입력하면 매칭 결과로 이동한다 | A-01 |

### 등록 규칙
- 기능 코드 항목은 page, component, hook, API 함수, GraphQL operation, Resolver, Service, Prisma 변경을 포함한다.
- 모든 기능 코드 항목은 대응 테스트 파일과 테스트 이름을 가져야 한다.
- 테스트 불가 항목은 범위 제외 또는 BLOCKED 사유를 명시한다.

### 예외 처리 테이블 형식
| 케이스 | 조건 | HTTP 상태 | 에러 코드 | 처리 |
|---|---|---|---|---|
| 인증 실패 | 토큰 만료 | 401 | UNAUTHORIZED | 리다이렉트 |

### 코드 예시
- TypeScript 기본
- 함수 시그니처 + 반환 타입 필수
- 구현부는 `...`으로 생략 (기획서이므로)

## 작업 큐 (tasks/)

### 작업 항목 형식
```markdown
### TASK-<번호>: <작업명>
- **상태**: PENDING | IN_PROGRESS | DONE
- **담당**: frontend | backend | designer
- **레포**: frontend | backend
- **우선순위**: HIGH | MEDIUM | LOW
- **의존성**: TASK-XXX (설명)
- **설명**: 작업 내용
- **완료 조건**:
  - 조건 1
  - 조건 2
- **참고 문서**: docs/... 경로
```

### 상태 관리
- 작업 시작 시 `PENDING` → `IN_PROGRESS`
- 완료 시 `IN_PROGRESS` → `DONE` (완료된 작업 섹션으로 이동)
